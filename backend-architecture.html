לפני בניית הבקאנד

קליינט מציג. בקאנד מחליט.

זהו מסמך החלטה לבניית בקאנד MVP: מה נשאר באפליקציה, מה עובר לשרת, אילו טבלאות ופעולות צריך, ומה לא בונים עדיין כדי לא להסתבך מוקדם מדי.

לבדיקת אישור ארכיטקטורת MVP כללית
REST API
Client UI, אנימציות, אלבום, בוחן מוצג, חוויית חנות
Backend מטבעות, פתיחת חבילה, RNG, בדיקת בוחן, בעלות
Node API קודם בונים שכבת API מקומית שניתן להעביר בהמשך ל-Railway.
שרת סמכותי הקליינט לא מחליט RNG, ניקוד, בעלות או מטבעות.
דאטה קיים כ-seed משתמשים ב-`game-data.json` כנקודת התחלה.
Supabase בהמשך המודל נכתב כך שיהיה קל להעביר ל-PostgreSQL/Auth.

חלוקת אחריות

העיקרון המנחה: כל מה שמשפיע על כלכלה, בעלות והתקדמות נשמר ומאומת בצד שרת.

גבול ברור
קליינט

מציג את האפליקציה, טוען נכסים, מנגן אנימציות, מציג שאלות ותשובות, ושולח פעולות לשרת.

Presentation
Backend API

מבצע פעולות משחק: קניית חבילה, פתיחת חבילה, יצירת ניסיון בוחן, בדיקת תשובות, חלוקת פרסים וקירור ניסיון.

Authority
Database

שומר משתמשים, פרופילים, יתרות, ledger, קלפים שנחשפו, קלפים שנאספו, ניסיונות בוחן ותוצאות.

State
Assets

תמונות קלפים וחבילות נשארות כקבצי WebP כרגע. בעתיד יעברו ל-Storage/CDN עם cache מסודר.

Storage later

זרימות משחק מול שרת

אלו הפעולות שה-MVP צריך כדי שהמשחק יפסיק להיות רק localStorage ויתחיל להתנהג כמו מוצר אמיתי.

MVP flows
1 טעינת משחק

הקליינט מבקש bootstrap: משתמש, יתרה, חנות, חבילות, אלבום ו-progress.

2 קניית חבילה

השרת בודק יתרה, מוריד מטבעות, יוצר רשומת רכישה ומחזיר חבילה זמינה לפתיחה.

3 פתיחת חבילה

השרת מחליט אילו 10 קלפים נחשפו ומסמן אותם כמועמדים, לא כקלפים שנאספו.

4 בחירת קלף

הקליינט מציג את הקלף מהאלבום. השרת בודק שהוא נחשף וזמין לבוחן.

5 ניסיון בוחן

השרת יוצר attempt, מחזיר 10 שאלות בלי לחשוף את `correct_index` לקליינט.

6 הגשת תשובות

השרת בודק תשובות, מחשב ציון, פותח בעלות ב-80%, או קובע retry למחר.

7 פרס ועדכון

השרת מעדכן collection, ledger, coins וסטטוס אלבום, ואז מחזיר state מעודכן.

8 סנכרון

הקליינט מרנדר את התוצאה בלבד. רענון עמוד לא יכול לזייף התקדמות.

API מוצע לשלב ראשון

השמות כאן הם חוזה עבודה. אפשר להתחיל מקומי, ואז למפות ל-Railway/Supabase בלי לשנות את הקליינט בצורה דרמטית.

Contract
Endpoint מי קורא מה הוא עושה
GET /api/bootstrap קליינט בטעינה מחזיר פרופיל, יתרה, חבילות, חנות, קלפים שנחשפו ונאספו.
GET /api/shop מסך חנות מחזיר קטלוג חנות: חבילות, מחיר, זמינות, מגבלות ומבצעי עונה.
POST /api/shop/purchase-pack חנות בודק יתרה, מוריד מטבעות, יוצר רכישה ומחזיר pack token לפתיחה.
POST /api/packs/open מסך חבילה מבצע פתיחה סמכותית, מסמן 10 קלפים כמועמדים ומחזיר reveal.
POST /api/cards/:cardId/quiz-attempts אלבום/בוחן פותח ניסיון בוחן אם הקלף נחשף, לא נאסף, ולא בקירור.
POST /api/quiz-attempts/:id/submit סיום בוחן בודק תשובות בצד שרת, מחזיר score, unlock, cooldown ופרסים.
GET /api/collection אלבום מחזיר מצב אלבום לפי חבילה: locked, discovered, owned, cooldown.

מודל נתונים ראשוני

זו סכמת MVP מינימלית. היא מספיק ברורה ל-Node מקומי, ומספיק קרובה ל-Supabase/PostgreSQL בהמשך.

Schema
content

seasons, packs, cards, questions, shop_items. רובו מגיע מ-`game-data.json` וקבצי questions.

Seed
accounts

users, family_profiles. ב-MVP אפשר להתחיל עם שחקן דמו אחד, אבל המבנה תומך בפרופילים.

Phase 2
economy

wallets, wallet_ledger, pack_purchases. כל שינוי מטבעות נרשם כאירוע, לא רק מספר סופי.

Required
collection

pack_openings, card_discoveries, collection_cards. מפריד בין קלף שנחשף לבין קלף שנאסף.

Required
quiz

quiz_attempts, quiz_attempt_answers. שומר ניסיון, שאלות שנשלחו, תשובות, ציון, pass/fail וזמן ניסיון הבא.

Required

כללי אבטחה וכלכלה

גם ב-MVP מקומי כדאי לבנות כאילו המשתמש מנסה לערוך את הדפדפן. זה יחסוך שכתוב גדול אחר כך.

Server first
לא שולחים תשובות נכונות

הקליינט מקבל אפשרויות להצגה בלבד. בדיקת נכונות נעשית בהגשה מול השרת.

חובה
אין מטבעות ב-localStorage

localStorage יכול לשמש cache תצוגה בלבד. היתרה האמיתית מגיעה מהשרת.

חובה
פתיחה אחת היא אירוע

כל קניית חבילה, פתיחת חבילה ובוחן נשמרים עם id כדי למנוע replay או כפילות.

חובה
קירור ניסיון בצד שרת

אם נכשלת בבוחן, השרת קובע מתי מותר לנסות שוב. שינוי שעון במכשיר לא אמור לעקוף זאת.

חובה
ממתין לאישור

מה אני מבקש לאשר

אם זה הכיוון, השלב הבא הוא לבנות backend MVP קטן שמחליף את ההחלטות המקומיות ב-API מסודר.

1
לבנות API מקומי קודם Node server עם endpoints למעלה, עדיין בלי תשלומים ובלי Auth מלא.
2
להשאיר קליינט קיים נחבר את `mvp.html` ל-API בהדרגה, בלי לזרוק את מה שעובד.
3
להשתמש בדאטה הקיים `game-data.json` יהיה seed ראשון לשרת ולחנות.
4
להכין מעבר ל-Supabase נכתוב storage adapter כדי שבהמשך נעביר state ל-PostgreSQL.

שלבי בנייה

מה נכנס ומה לא נכנס לשלב הראשון.

Phase 1: Backend MVP API, state, wallet, shop placeholder, pack open, quiz submit, collection sync.
Phase 2: Persistence אמיתי SQLite או Supabase, migrations, seed scripts, בדיקות API.
Phase 3: Accounts Auth, parent account, family profiles, sync בין מכשירים.
לא עכשיו תשלומים אמיתיים, marketplace, admin מלא, Three.js production, PWA אופליין מלא.

מסמך זה נועד לאישור לפני בניית בקאנד. אחרי אישור, המימוש הראשון יהיה קטן, מדיד, ויחליף בהדרגה את הלוגיקה המקומית ב-`mvp.html`.
